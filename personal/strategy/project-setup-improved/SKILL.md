# プロジェクトセットアップスキル

## ゴール

「Claudeがプロジェクトを深く理解し、自律的かつ的確に動けるようにする」こと。
ディレクトリ構成の設計/整理と、CLAUDE.mdの生成/更新を**対話を重ねながら段階的に**行う。

## なぜ対話的にやるのか

CLAUDE.mdはClaudeにとっての地図であり行動指針書。精度が高いほど的確に動ける。
しかし、ユーザーの頭の中にある「こう動いてほしい」を一発で言語化してもらうのは難しい。

だからこそ、こちらから質問を投げかけ、選択肢を提示し、提案して反応をもらう
**対話のキャッチボール**が重要になる。ヒアリングシートを埋めるのではなく、
会話の中で自然にプロジェクトの全体像と行動戦略を浮かび上がらせることを目指す。

---

## 全体の流れ（4フェーズ）

各フェーズの最後に理解した内容をまとめて提示し、ユーザーの承認を得てから次に進む。

```
フェーズ0: 初動判断（自動 — 環境検出 + 新規/既存判定）
    ↓
フェーズ1: プロジェクトを知る（現状把握 + 目的・チーム・課題の対話）
    ↓ 理解メモの確認
フェーズ2: 戦略を決める（Claudeの役割・ペルソナ・行動指針・優先順位の対話）
    ↓ 行動戦略の確認
フェーズ3: 形にする（ディレクトリ構成の対話的ネゴシエーション + CLAUDE.md生成）
    ↓ 最終確認
```

---

## フェーズ0: 初動判断（自動）

ユーザーとの対話を始める**前に**、以下を自動で判定する。

### 0-1. 対象ディレクトリの特定

ユーザーがパスを指定していればそこを使う。指定がなければ：
- Cowork: マウントされたワークスペースフォルダを使う
- Claude Code: カレントディレクトリを使う

### 0-2. 新規 vs 既存の判定

```bash
# ファイル数で判定
file_count=$(find . -maxdepth 1 -not -name '.' -not -name '.git' -not -name '.gitignore' | wc -l)
```

- ファイル数が2以下 → **新規プロジェクト**として進める
- ファイル数が3以上 → **既存プロジェクト**として進める

### 0-3. 既存CLAUDE.mdの有無

既存のCLAUDE.mdがあれば「更新モード」、なければ「新規作成モード」。
更新モードの場合、現在のCLAUDE.mdの内容を読み込んでおく。

**ここまでは無言で実行する。** 結果はフェーズ1の冒頭でユーザーに自然に共有する。

---

## フェーズ1: プロジェクトを知る

### 1-1. 技術的な現状把握（自動分析）

**既存プロジェクトの場合**、以下を自動で分析する：

```bash
# ディレクトリ構造（2階層）
find . -maxdepth 2 -type f | head -50
# 設定ファイル
cat package.json 2>/dev/null || cat Cargo.toml 2>/dev/null || cat pyproject.toml 2>/dev/null || cat go.mod 2>/dev/null
# 既存のCLAUDE.md
cat CLAUDE.md 2>/dev/null
# git設定
cat .gitignore 2>/dev/null | head -20
# READMEの冒頭
head -30 README.md 2>/dev/null
```

分析から判定する：
- 技術スタック（言語、フレームワーク、ビルドツール）
- プロジェクトの種類（Webアプリ、API、ライブラリ、ドキュメント、混合）
- ディレクトリ構成の整理度
- 既存CLAUDE.mdの品質（あれば）

**新規プロジェクトの場合**は分析をスキップして、対話から始める。

### 1-2. 最初の対話 — プロジェクトの全体像

分析結果があればそれを共有しつつ、対話を始める。
**1回の質問で聞きすぎない**のが重要。最初の質問は1つだけ。

既存プロジェクトの場合：
```
[分析結果を簡潔に共有]

まず教えてください — このプロジェクトを一言で説明するとしたら？
```

新規プロジェクトの場合：
```
新しいプロジェクトですね。まず教えてください — 何を作ろうとしていますか？
```

AskUserQuestionを使うタイミングの判断基準：
- **選択肢が明確なとき** → AskUserQuestion（選択式）を使う
- **自由に語ってほしいとき** → AskUserQuestion（自由記述）を使う
- **会話の流れで自然に聞けるとき** → 普通のメッセージで聞く

### 1-3. 掘り下げ対話 — チーム・ステークホルダー・課題

最初の回答を受けて、さらに掘り下げる。ここでは**状況に応じて**以下の観点を聞く。
全部聞く必要はない — プロジェクトの種類に応じて重要なものを選ぶ。

**聞くべき観点（優先度順）：**

1. **あなたの役割**: このプロジェクトでどういう立場か
   - 例: 個人開発、テックリード、PM、CTO、コンテンツ管理者
   - AskUserQuestion（選択式 + 「その他」）が効果的

2. **チーム構成**: 一人でやっているか、チームか
   - チームの場合: 何人か、どんな役割分担か
   - Claudeに「チームの誰か」的な役割を期待しているか

3. **プロジェクトの現在地**: どのフェーズか
   - AskUserQuestion で選択式が効果的：
     - アイデア段階（まだコード書いてない）
     - 開発初期（基盤を作っている）
     - 開発中期（機能を増やしている）
     - 運用中（ユーザーがいる）
     - リファクタリング/移行中

4. **今の課題・痛み**: 何に困っているか、何を改善したいか
   - 「Claudeにこのプロジェクトをセットアップさせようと思った理由は？」と聞くと本質的な課題が出やすい

5. **ビジネスコンテキスト**（該当する場合）:
   - 誰のためのプロダクトか
   - KPIや成功指標はあるか
   - 期限やマイルストーンはあるか

**対話の進め方のコツ：**
- 回答から推測できることはこちらから提示して確認する（「〜ということは、Xですか？」）
- 長い回答が返ってきたら、要点を整理して返す
- 質問は1ターンにつき1-2個に抑える
- ユーザーが「あんまり決まってない」と言ったら深追いしない

### 1-4. 理解メモの提示と確認

対話で得た情報をまとめて提示する。このまとめ自体が次のフェーズへの布石になる。

```
📋 プロジェクト理解メモ
━━━━━━━━━━━━━━━━━━━━━━━━━━
■ プロジェクト: [名前] — [一言の説明]
■ 種類: [Webアプリ / API / ライブラリ / コンテンツ / etc.]
■ 技術スタック: [言語、フレームワーク、主要ツール]
■ あなたの役割: [個人開発 / テックリード / etc.]
■ チーム: [一人 / N人チーム（役割概要）]
■ 現在地: [開発初期 / 運用中 / etc.]
■ 課題・目的: [なぜこのセットアップをしようとしているか]

この理解で合っていますか？修正があれば教えてください。
```

**確認が取れたらフェーズ2に進む。修正があれば対話を続ける。**

## フェーズ2: 戦略を決める

ここがこのスキルの核心。「Claudeがこのプロジェクトでどう動くべきか」を対話で決める。
フェーズ1で得た情報をもとに、**こちらから提案して擦り合わせる**スタイルで進める。

### 2-1. Claudeの役割を定義する

ユーザーの立場とプロジェクトの性質を踏まえて、まずこちらから役割を提案する。

**提案のテンプレート：**

```
[プロジェクトの状況]を踏まえて、Claudeの役割として以下を提案します：

🎭 提案する役割: [役割名]
例えば…
- 「シニアエンジニアとしてペアプロ」
- 「テクニカルライターとしてドキュメント整備」
- 「プロジェクトマネージャー補佐として整理・管理」
- 「フルスタック開発のソロパートナー」

この方向で合っていますか？それとも別のイメージがありますか？
```

**役割の具体的な選択肢（プロジェクト種類ごと）：**

| プロジェクト種類 | 提案する役割の例 |
|---|---|
| 個人開発のWebアプリ | フルスタック開発パートナー（設計〜実装〜テスト） |
| チーム開発のAPI | シニアバックエンドエンジニア（実装 + コードレビュー） |
| ライブラリ/OSS | メンテナ補佐（PR対応、ドキュメント、テスト） |
| コンテンツサイト | テクニカルライター + サイト管理 |
| データ分析 | データエンジニア（パイプライン構築 + 分析） |
| 既存コードのリファクタ | リファクタリング専門家（段階的改善提案 + 実装） |

### 2-2. Claudeのコミュニケーションスタイル

役割が決まったら、**どう対話するか**も決める。
ここは軽くAskUserQuestionで聞くのが効果的。

```
Claudeのコミュニケーションスタイルはどれが近いですか？

A) 簡潔重視 — コードと結果で語る。説明は聞かれたときだけ
B) バランス型 — 要点を説明しつつ、手を動かす
C) 丁寧解説型 — なぜそうするかを毎回説明。学習目的にも使いたい
D) コーチ型 — 答えを直接出すより、考え方のヒントを出す
```

また、以下も必要に応じて聞く：
- **言語**: 日本語 / 英語 / コードは英語でコメントは日本語 / etc.
- **確認頻度**: こまめに確認してほしい / まとめて一気にやってほしい
- **レスポンス量**: 短く / 必要十分 / 詳しく

### 2-3. 行動指針と自律度

ここも**提案型**で進める。プロジェクトの現在地から適切なデフォルトを提案する。

**提案のパターン：**

開発初期の場合:
```
開発初期なので、スピード重視の方針を提案します：

■ 自律度: 高め（基本は自分で判断して進める）
  - 新しいファイル/ディレクトリの作成: OK
  - 既存コードのリファクタ: 小規模ならOK、大規模は確認
  - 依存ライブラリの追加: 提案してから

■ テスト方針: 主要パスのみ（網羅性は後から）
■ コード品質: 動くことが最優先。完璧さは後から
■ やらないこと: [プロジェクト固有のNG事項があれば]

この方針でいいですか？調整したいところがあれば教えてください。
```

運用中の場合:
```
ユーザーがいる運用中のプロジェクトなので、安全重視の方針を提案します：

■ 自律度: 中程度（既存を壊す可能性がある変更は確認）
  - バグ修正: 自分で判断してOK
  - 機能追加: 設計を確認してから実装
  - DB変更: 必ず確認
  - 破壊的変更: 絶対に確認

■ テスト方針: 必須（変更箇所 + 影響範囲のテスト）
■ コード品質: 既存コードのスタイルに合わせる
■ やらないこと: 本番環境への直接操作、データ削除

この方針でいいですか？
```

### 2-4. 作業ルールの確認

プロジェクトの種類に応じて、必要なルールを確認する。
全部聞く必要はない — 該当するものだけ聞く。

**コードプロジェクトの場合:**
- コミットメッセージの規約（Conventional Commits / 自由 / 日本語OK）
- ブランチ戦略（feature branch / trunk-based / 直push）
- テストフレームワークと実行コマンド
- リンター/フォーマッターの設定
- CI/CDの有無と確認事項

**コンテンツプロジェクトの場合:**
- コンテンツのフォーマット（Markdown / MDX / etc.）
- 命名規則（ファイル名、ディレクトリ名）
- 画像やアセットの管理方法
- 公開フロー

**自動検出できるものは聞かない:**
`package.json` の `scripts` セクション、`.eslintrc`、`prettier.config` 等から
読み取れるルールは自動で拾い、確認だけする。

### 2-5. 行動戦略の確認

フェーズ2の対話結果をまとめて提示する。

```
🎯 Claudeの行動戦略
━━━━━━━━━━━━━━━━━━━━━━━━━━

■ 役割: [定義した役割]
■ コミュニケーション: [スタイル + 言語]
■ 自律度: [高/中/低 + 具体的な境界線]

■ やること:
  - [主な責務1]
  - [主な責務2]
  - [主な責務3]

■ やらないこと:
  - [明示的なNG事項]

■ 作業ルール:
  - テスト: [方針]
  - コミット: [規約]
  - コード品質: [基準]

■ 判断基準:
  - 迷ったら: [方針]
  - 優先順位: [1. xxx  2. yyy  3. zzz]

この戦略で進めてよいですか？
```

**確認が取れたらフェーズ3に進む。**

## フェーズ3: 形にする

### 3-1. ディレクトリ構成の対話的ネゴシエーション

**新規プロジェクトの場合:**

フェーズ1・2の情報をもとに構成を**提案**する。
提案はtree形式で見せ、各ディレクトリの役割を簡潔に説明する。

```
プロジェクトの内容を踏まえて、以下の構成を提案します：

project-root/
├── CLAUDE.md          # Claudeの行動指針
├── README.md          # プロジェクト説明
├── src/               # ソースコード
│   ├── app/           # アプリケーション本体
│   ├── lib/           # 共通ライブラリ
│   └── types/         # 型定義
├── tests/             # テスト
├── docs/              # ドキュメント
├── scripts/           # ユーティリティスクリプト
└── .claude/           # Claude関連設定
    └── skills/        # プロジェクト固有スキル

変更したいところはありますか？
例えば「docs/は要らない」「components/を追加したい」など。
```

ユーザーのフィードバックを受けて修正 → 再提示 → 合意のサイクルを回す。

**既存プロジェクトの場合:**

現状の構成を分析し、**改善提案がある場合のみ**提案する。
改善がなければ「現状の構成で問題ないので、そのまま進めます」と伝える。

改善提案の例：
```
現状の構成を見ました。いくつか提案があります：

1. [追加] docs/ — 設計ドキュメントの置き場所がないので追加を提案
2. [追加] .claude/skills/ — プロジェクト固有のスキルを入れる場所
3. [変更なし] src/, tests/ — 現状の構成で問題なし

1と2を追加してよいですか？他に整理したいところはありますか？
```

**重要: 既存プロジェクトの構成を勝手に大きく変更しない。**
提案はあくまで提案。ユーザーの明示的な合意なしに実行しない。

### 3-2. CLAUDE.mdの生成

フェーズ1・2の対話結果をCLAUDE.mdに落とし込む。

#### 必須セクション（全プロジェクト共通）

```markdown
# CLAUDE.md

## プロジェクト概要
[プロジェクトの目的、技術スタック、現在のフェーズを1-3文で]

## Claudeの役割
[フェーズ2で決めた役割定義]
- 役割: [何として動くか]
- コミュニケーション: [スタイル + 言語]
- 自律度: [高/中/低 + 境界線]

## 行動指針
[フェーズ2で合意した行動戦略]
- やること: [主な責務]
- やらないこと: [明示的なNG]
- 迷ったら: [判断基準]

## 優先順位
1. [最優先]
2. [次に重要]
3. [意識すること]

## コマンド
[開発・ビルド・テスト等の主要コマンド。コピペで動く形式]
```

#### 追加セクション（プロジェクトの種類と対話内容に応じて選択）

```markdown
## アーキテクチャ
[技術スタック、ディレクトリ構成、主要な設計判断]

## Git運用
[ブランチ戦略、コミットメッセージ規約、PRのルール]

## コーディング規約
[フォーマッタ設定、命名規則、コンポーネント構成]

## テスト方針
[テストの書き方、カバレッジ目標、テスト実行方法]

## コンテンツ管理
[コンテンツの置き場所、公開フロー、トンマナ]

## チーム情報
[チーム構成、役割分担、Claudeとの協業パターン]

## 外部ツール連携
[デプロイ先、アナリティクス、外部API]

## ビジネスコンテキスト
[プロジェクトの背景、KPI、戦略情報への参照]
```

#### CLAUDE.md作成のガイドライン

- **簡潔に**: 長すぎるCLAUDE.mdはかえって邪魔。各セクションは必要十分に絞る
- **コマンドは実行可能な形で**: `pnpm run dev` のようにコピペで動く形式
- **パスは具体的に**: 「設定ファイル」ではなく `tailwind.config.ts`
- **行動指針は判断可能な基準で**: 「丁寧にやる」→「テストカバレッジ80%以上を目標」
- **外部参照を活用**: 巨大な情報はCLAUDE.md本体に書かず `@path/to/file.md` で参照
- **更新しやすい構造**: セクション構成を明確にし、部分更新が容易にする
- **対話で出た生の言葉を活かす**: ユーザーが使った表現をそのまま使う方が、後で読み返したとき意図が伝わる

### 3-3. CLAUDE.mdの対話的レビュー

生成したCLAUDE.mdをユーザーに提示する。
ただし「これでいいですか？」だけでなく、**具体的なレビューポイント**を示す。

```
CLAUDE.mdを作成しました。以下のポイントを確認してください：

1. 「Claudeの役割」セクション — [要約]。この定義で合っていますか？
2. 「行動指針」の自律度 — [要約]。ここの塩梅は適切ですか？
3. 「コマンド」セクション — 実際に動くか確認をお願いします

修正したい箇所があれば教えてください。
```

### 3-4. 適用と完了

1. ユーザーの確認が取れたらファイルを書き込む
2. ディレクトリの追加がある場合は作成する
3. 最後に完了メッセージを出す：

```
セットアップ完了です。

📝 作成/更新したもの:
- CLAUDE.md（Claudeの行動指針）
- [作成したディレクトリがあれば]

次にClaudeがこのプロジェクトで作業するとき、
CLAUDE.mdを読んで自律的に動けるようになっています。

何か追加で調整したいことがあれば、いつでも言ってください。
```

## 既存のCLAUDE.mdを更新する場合

既存のCLAUDE.mdがある場合の特別なフロー。ゼロから書き直すのではなく差分で改善する。

### 更新フローの手順

1. 現在のCLAUDE.mdを読み、ユーザーに内容を簡潔に共有する
2. 以下を確認する：
   - 「役割」「行動指針」セクションがあるか → なければ追加を強く提案
   - 古くなっている情報はないか（コマンド変更、構成変更など）
   - 不足しているセクションはないか
3. 改善案を**追加・変更・削除**に分けて提示する
4. フェーズ2の戦略対話は**既存CLAUDE.mdがある場合でも実施する**
   - 特に「役割」「行動指針」がないCLAUDE.mdは「地図はあるが目的地がわからない」状態
   - 対話を通じてこれらを追加することで、Claudeの自律性が大幅に向上する
5. 確認を得てから更新を適用する

## 対話設計の原則

このスキル全体を通じて、以下の対話原則を守る。

### 1. 1ターン1-2質問

質問が多すぎると対話が面接のようになる。1ターンの質問は1-2個に抑える。
複数の観点を聞きたいときは、最も重要なものから順に、ターンを分けて聞く。

### 2. 提案ファースト

「どうしますか？」と白紙で聞くのではなく、「Xを提案します。どうですか？」と聞く。
ユーザーは「はい」「いいえ」「ここを変えて」で答えられるので負担が少ない。

### 3. 推測してから確認

対話で得た情報から推測できることは先にこちらから提示して確認する。
「TypeScript + Next.jsということは、Vercelにデプロイする予定ですか？」のように。

### 4. 深追いしない

ユーザーが「あんまり決まってない」「わからない」と言ったら、そこはデフォルト値で進める。
「まだ決まっていないなら、一旦こうしておきます。後で変更できます」が正解。

### 5. AskUserQuestionの使い分け

| 状況 | 方式 |
|---|---|
| 選択肢が明確（3-5個） | AskUserQuestion 選択式 |
| 自由に語ってほしい | AskUserQuestion 自由記述 |
| 会話の流れで自然に聞ける | 通常メッセージ |
| Yes/No確認 | 通常メッセージ |
| まとめの確認 | 通常メッセージ（確認フォーマットで） |

### 6. 「何も決まっていない」場合のデフォルト

ユーザーが詳細を決めていない場合のデフォルト値を持っておく：

| 項目 | デフォルト |
|---|---|
| 役割 | フルスタック開発パートナー |
| コミュニケーション | バランス型 |
| 自律度 | 中（既存を壊す変更は確認） |
| テスト | 主要パスのみ |
| コミット | Conventional Commits |
| 言語 | ユーザーの使用言語に合わせる |
