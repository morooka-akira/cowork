# Notion可視性設計書（Timelab teamspace）

作成日：2026-04-23
対象：Timelab teamspace 配下のすべてのページ・DB
作成者：諸岡 明（秘書AIと協働）

---

## 1. 背景と目的

Timelabのメンバーには「正社員・アルバイト」と「業務委託・副業メンバー」が混在しており、
現状のNotion運用では両者が同じアクセス権限で動いている部分が多い。
一方で、**業務委託・副業メンバーに見せるべきでない情報**（採用・評価・財務・戦略など）が
Timelab teamspace 配下に散在している。

本書は、Timelab teamspace 配下のページ/DBを 3 段階の可視性に整理するための
ルール・現状Gap・推奨アクションをまとめたもの。

なお、他のteamspace（Board・hamamatsu・TagoOne・規約）は別途 private 化する方針で
既に方向性が固まっているため、本書のスコープからは除外する。

---

## 2. 登場する「人」の定義

| カテゴリ | 定義 | 例 |
|---|---|---|
| **社員・アルバイト**（= 内部） | 雇用契約を結んでいるメンバー | 保積、高橋、藤村など |
| **業務委託・副業**（= 外部） | 業務委託契約でスポット/パートタイムで関わるメンバー | KeitaさんなどBiz/Dev側の外部人材 |
| **Guest** | 案件単位でNotionに招待する社外の人 | クライアント、候補者など |

本書で「外部」と書いた場合、基本は **業務委託・副業メンバー** を指す。

## 3. 3 段階の可視性ルール

### 🟢 Tier-A：全公開（業務委託を含む全員OK）
- **ルール**：Timelab という会社・プロダクト・文化を理解するのに必要な「外に出ても困らない」情報
- **判定基準**：
  - 固有の個人情報（評価・給与・1on1内容）を含まない
  - 具体的な財務数値を含まない
  - クライアントごとの秘匿条件（NDA対象）を含まない
  - プロダクトの公開情報・ブランディング・文化に属する
- **運用**：Timelab teamspace の default 閲覧権限。`Can view` を基本。

### 🟡 Tier-B：社員・アルバイトのみ（業務委託はNG）
- **ルール**：組織運営・採用・評価・財務の詳細など、雇用している人間だけに開示すべき情報
- **判定基準**：
  - 個人の評価・1on1・処遇が書かれている
  - 採用候補者の固有名・提示条件・内部評価が書かれている
  - 月次/四半期の財務数値、資金繰り、銀行交渉
  - 会社全体の戦略方針のうち、数値目標や採用計画を含むもの
- **運用**：社員・アルバイトグループにのみ `Can view` / `Can edit`。業務委託には非公開。

### 🔴 Tier-C：限定共有（プロジェクト・役職の指名メンバーのみ）
- **ルール**：さらに狭く、特定プロジェクト or 特定ロール（経営・人事・責任者）だけに開示
- **判定基準**：
  - 経営会議の議事録／Board 提出資料
  - 資金調達・M&A・重要契約の交渉記録
  - 特定プロジェクトの技術仕様でクライアント NDA 対象
  - 採用オファー・給与改定の個別判断記録
- **運用**：個別にメンション / Member リストで指定。親ページから権限を切って独自権限を付与。

- **採用定例（MTG Note DB 内の複数ページ）**：候補者の実名、面接評価、提示時給、想定年収などが書かれている。例：`33caf136-accb-8041-878d-c62fe39f22f8`（東度さん・永村さんの評価と条件）

### 5.2 経営・財務

## 6. 現状 Gap（想定される問題点）

1. **Timelab Inc. ポータルページ自体が「誰でも自由にアクセス」と謳っている**
   → 業務委託を teamspace に招待した瞬間、上記すべての NG 情報にフルアクセスできてしまう。
   → ポータルに書かれた「すべての情報に誰でもアクセス」の文言は **社員・アルバイトのみ** を前提とした古いものと思われる。更新必要。

2. **MTG Note DB が「全 MTG 議事録の一元管理」になっている**
   → 採用定例と全社定例が同じ DB に入っており、**DB 単位で権限を切れない**。
   → ページ単位または「タグ」によるビュー分離 + 権限管理が必要。

3. **個人ページが teamspace 配下にある**
   → 諸岡の個人ページに「諸岡タスクDB」が存在し、そこに評価系タスクが混在。
   → 個人ページは各自のプライベート配下に移すか、権限を個別に切る必要あり。

4. **業務委託メンバーが Timelab teamspace のメンバーになっている（想定）**
   → Notion の teamspace メンバー構造上、teamspace メンバーはその下の「open」ページを基本閲覧できてしまう。権限モデルの見直しが要る。

## 7. 推奨アクション（優先度順）

### P0（今週中にやるべき）
1. **MTG Note DB の権限を見直し**
   - 現状：全員閲覧可能と想定
   - 推奨：DB を「全社 MTG Note DB（Tier-A/B）」と「限定 MTG Note DB（Tier-C：採用・経営・1on1）」に分ける
   - または、1つの DB のままで「種別」プロパティを使い、**DB 権限を Tier-B に制限 → 一部ページを個別に Tier-A で公開** という運用に変更
   - 即対応：採用定例・経営会議・1on1 ページだけでも個別権限を `noaccess` に変更して業務委託から隠す

2. **FY26経営方針ページの権限を社員のみに変更**
   - 親の「戦略方針」ページを Tier-B に切り替え

3. **個人ページ配下の評価・1on1 タスクを本人+マネージャーに制限**
   - 諸岡タスクDB の該当ページに個別権限設定

### P1（今月中）
4. **Timelab teamspace のメンバー構成を棚卸し**
   - 誰が社員・アルバイトで、誰が業務委託か、Notion のグループ機能で分離
   - `社員・アルバイト` グループと `業務委託` グループを作成

5. **ポータルページ（Timelab Inc.）のキャッチコピーを現実に合わせて修正**
   - 「すべての情報に誰でも自由にアクセス」は現実とズレているので表現を変える

6. **個別DB（OKR / Decision / Roadmap / KPT）の権限を Tier-B に変更**

7. **Lynx / Calico / Penguin Studio の各プロジェクトページ配下を点検**
   - プロジェクトに関与している業務委託には個別にページ単位で権限付与、関与していない業務委託にはトップから隠す

### P2（四半期単位で継続）
8. **新規ページ作成時の「どこに作るか」ルール化**
   - CLAUDE.md（または社内 Notion ポリシー）に「新規ページを作るときは Tier を意識し、Tier-B 以上なら専用の親ページ配下に作る」というルールを追加

9. **定期棚卸し**
   - 四半期に1回、Timelab teamspace 配下の新規ページを可視性チェック

## 8. 運用ルール（新規作成時のフロー）

新しいページ/DB を作るときに迷わないように、以下をフローチャート化：

```
新しいページを作る
    ↓
Q1: このページに以下のいずれかが含まれるか？
    - 個人の固有名を含む評価・1on1・処遇
    - 具体的な財務数値・資金調達情報
    - 採用候補者の実名と条件
    - NDA対象のクライアント情報
    ↓ Yes → 🔴 Tier-C の親ページ配下に作る（例：経営・Board 配下）
    ↓ No
Q2: 雇用している人だけに開示すべき情報か？
    - 組織運営の詳細
    - 戦略の数値目標
    - 会社の KPI / KR
    ↓ Yes → 🟡 Tier-B の親ページ配下に作る（例：戦略方針 / 個人ページ配下）
    ↓ No
    → 🟢 Tier-A の場所に作ってOK（Timelab ポータル直下）
```
