# スタジオテックリード候補者スクリーニング

リクルートマーカー（https://app.recruit-marker.jp/）上の候補者リストに対して、
LinkedIn / YouTrust のプロフィール情報をもとに採用要件チェックを行い、
要件に合致しない候補者を理由付きでリストアップするスキル。

## 前提条件

- Chrome ブラウザが利用可能であること（Claude in Chrome の MCP ツールを使用）
- リクルートマーカーにログイン済みの状態であること
- LinkedIn にもログイン済みであることが望ましい（プロフィールの閲覧制限を緩和できる）

## ワークフロー概要

```
1. リクルートマーカーにアクセス → ログイン確認
2. 作業リストの特定（URLまたはリスト名）
3. リストのグリッドからLinkedIn/YouTrustリンクを収集
4. 各候補者のプロフィールを閲覧 → チェック要件で判定
5. 結果をマークダウンファイルに出力
6. Slackで完了通知
```

---

## Step 1: リクルートマーカーへのアクセスとログイン確認

Chrome ブラウザツールで対象のリストURLに直接アクセスする。
ユーザーがURLを指定していれば（例: `https://app.recruit-marker.jp/list/person-list/62064`）そのまま使う。

### ログイン状態の確認

ページを開いたらまずログイン状態を確認する。リスト画面が表示されればログイン済み。
ログイン画面にリダイレクトされた場合は、ユーザーに再ログインを依頼する。

---

## Step 2: 作業リストの特定

ユーザーがリストURLやリスト名を指定していれば、そのリストに移動する。
指定がない場合はユーザーに「どのリストで作業しますか？」と確認する。

---

## Step 3: リストからリンク情報の収集

### リクルートマーカーのUI構造

リスト画面はグリッド（テーブル）形式で候補者が表示される。
`read_page` ツールで `filter: "interactive"` を指定すると、各候補者の行に含まれるリンクが取得できる。

各候補者のグリッド行には以下のリンクが含まれる（候補者によって異なる）：
- Google検索リンク（`google.co.jp/search?q=名前+会社名`）
- **LinkedInリンク**（`linkedin.com/in/...`）
- **YouTrustリンク**（`youtrust.jp/users/...`）
- FacebookやGitHubリンク（ある場合）

表示件数を100件に設定して一覧を取得する。64件以上ある場合はページネーションで次ページも処理する。

### リンクの収集手順

1. `read_page` で interactive 要素を取得
2. 各候補者のLinkedIn/YouTrustリンクをテキストから抽出
3. 候補者名はリクルートマーカーのリスト表示テキスト（`get_page_text`）から取得
4. **リクマのリスト表示テキストから、各候補者の「現職の会社名」と「現職の役職タイトル」を必ず記録する。** これは後の判定で重要な情報源となる。
5. 各候補者のリクルートマーカー上の個人ページURL（`https://app.recruit-marker.jp/list/person-list/{リストID}/person/{personID}`）も収集する。グリッド行の候補者名リンクから取得できる

---

## Step 4: 候補者の順次プロフィール確認

各候補者のプロフィールを1人ずつ確認していく。
新しいタブで各プロフィールを開き、情報を取得する。

### プロフィール情報源の優先順位と活用方針

**【最重要】リクルートマーカーのリスト上の情報（現職タイトル・会社名）は常に判断材料として活用する。**
リクマのリスト表示テキストには候補者の現職の会社名と役職が表示されている。
LinkedInが3次つながりで情報制限されていても、リクマの情報だけで判定できるケースが多い。
LinkedInヘッドラインがリクマ情報と矛盾する場合、リクマの方がソーシング時点の最新情報である可能性が高いため、**リクマ情報を優先する。**

**YouTrustを最優先で使う。** テスト結果から、YouTrustはLinkedInより圧倒的に情報が豊富であることがわかっている：
- 職歴の会社名・役職・期間が詳細に取得可能
- スキル・経験が明記されている
- 自己紹介テキストに技術スタックが書かれていることが多い

**LinkedInは補助的に使う。** LinkedInはつながり度合いによって表示内容が制限される：
- 3次以上のつながりの場合、職歴セクションが表示されないことがある
- ヘッドライン（名前の下の1行テキスト）にはスキルや技術名が含まれることが多く有用
- 所在地・学歴は比較的表示されやすい
- **ヘッドラインに役職（Manager, Lead, Architect, Engineering Manager等）が含まれていれば、それだけで要件3の判定材料として十分**

**情報収集の戦略：**
1. YouTrustリンクがある → YouTrustを開いて `get_page_text` で情報取得
2. YouTrustがなくLinkedInのみ → LinkedInを開いてヘッドライン・表示可能な情報を取得
3. LinkedInが3次つながりで情報制限されている場合 → **リクマのリスト上の情報（現職タイトル・会社名）とLinkedInヘッドラインを組み合わせて判定を行う。** これだけでも多くの場合は判定可能。
4. どちらもない → リクルートマーカーのリスト上の情報のみで判定を試みる。
5. 「判定保留」は、リクマ情報・LinkedInヘッドラインいずれからも職種や経験レベルが全く読み取れない場合にのみ使う（最終手段）

### プロフィールの読み取り方法

**YouTrustの場合：**
```
get_page_text でプロフィール全体を取得。以下の情報が含まれる：
- 自己紹介（Skills, 技術スタック）
- 職歴・学歴（会社名、役職、期間が時系列で表示）
- スキルタグ
```

**LinkedInの場合：**
```
get_page_text でヘッダー情報を取得。取得できる情報：
- 名前
- ヘッドライン（現職＋スキル名が含まれることが多い）
- 所在地
- 学歴（表示される場合）

職歴が表示されない場合は、スクロールして確認を1回試みる。
それでも表示されなければ、ヘッドラインの情報で判定する。
```

### レート制限への配慮

- プロフィール間で2〜3秒の `wait` を入れる（特にLinkedIn）
- LinkedIn のアクセス制限に引っかかった場合（CAPTCHA等）はユーザーに報告
- 一度に大量のLinkedInプロフィールを連続で開かない

## Step 5: チェック要件による判定

取得したプロフィール情報を以下の4要件に照らして判定する。

### 要件1: 日本国籍であること

以下の情報から総合的に判断する：
- 名前が日本人名であるか（漢字名、日本語のカタカナ/ひらがな表記など）
- 所在地が日本であるか
- 学歴が日本の教育機関か
- プロフィールが日本語で書かれているか

明らかに日本国籍でないと判断できる場合のみ「不合格」とする。
判断が微妙な場合は「判断保留」として記録し、理由を付記する。

### 要件2: 日本人であること

要件1と重なる部分もあるが、以下の観点で補強する：
- 日本語でのコミュニケーションが問題なくできると推定できるか
- プロフィールの言語・文体が日本語ネイティブと判断できるか
- 日本の企業での勤務経験があるか

要件1と合わせて総合的に判断し、明らかに該当しない場合のみ「不合格」とする。

### 要件3: チームやプロジェクトをリードできる経験

職歴やプロフィール情報から、プロジェクトやチームをリードした経験があるかを確認する。

**重要：クライアントワーク（受託開発）に限定しない。** プロダクト企業でのEM/TL経験も十分にリード能力の証明となる。
重要なのは「チームをリードしてプロジェクトを推進する能力」があるかどうかであり、それがクライアントワークかプロダクト開発かは問わない。
クライアントワーク経験があれば「なお良い」（加点要素）として扱う。

**合格の指標（以下のいずれかに該当すれば合格）：**
- エンジニアリングマネージャー（EM）の役職経験
- テックリード（TL）、リードエンジニアの役職経験
- プロジェクトマネージャー（PM）の役職経験
- Senior Software Engineer 以上の役職で、チームリード的な役割を担っている
- 「本部長」「部長」「マネージャー」などの管理職経験
- 受託開発・クライアントワークにおける顧客折衝・要件定義の経験（加点要素）
- 提案書作成、見積もり、技術選定に関わった経験（加点要素）

**不合格の指標：**
- 個人開発やジュニアメンバーレベルの経験のみでリード経験が見当たらない
- 役職名に Lead, Manager, Senior, Architect, 本部長, 部長 等のリーダーシップを示す語句が一切含まれない
- コミュニケーションに関する役割が一切見当たらない

**判定のポイント：**
- LinkedInヘッドラインやリクマの役職に「Manager」「Lead」「Architect」「EM」「本部長」「部長」等が含まれていれば、それだけで合格と判断してよい
- プロダクト企業（Rakuten、ZOZO、Treasure Data等）であってもリード職であれば合格
- SIer・受託企業でのリード経験があればさらに高評価

### 要件4: バックエンドのスキルが一定レベルあり、アプリケーション開発の初期設計ができること

プロジェクト立ち上げ時に、バックエンドのアーキテクチャ設計・技術選定を
主導できるレベルのスキルがあるかを確認する。

**前提条件：アプリケーションエンジニアであること。**
SRE、インフラエンジニア、プラットフォームエンジニアなど、アプリケーション開発を主業務としない職種は、
バックエンド初期設計の要件を満たさないため **不合格** とする。

**合格の指標：**
- バックエンド言語/フレームワークの実務経験が3年以上（Ruby/Rails, Python/Django/Flask, Node.js/Express, Go, Java/Spring, PHP/Laravel 等）
- データベース設計の経験（MySQL, PostgreSQL, MongoDB 等のスキーマ設計・最適化）
- API設計・開発の経験（REST API, GraphQL 等）
- インフラ/クラウド設計の経験（AWS, GCP, Docker, Kubernetes 等）
- システムアーキテクチャの設計・技術選定に携わった経験
- 要件定義からの技術的な落とし込み・設計ドキュメント作成の経験
- LinkedInヘッドラインやリクマの役職名に「Application Engineer」「Software Engineer」「Backend Engineer」「Architect」「Full Stack」等が含まれていれば、アプリケーション開発者と判断してよい

**プロフィール・スキル欄の具体的な技術スタックからの判断：**
YouTrustのスキルタグやLinkedInのスキル欄に記載されている具体的な技術名を重視して判定する。
- バックエンド系技術（Ruby, Python, Go, Java, PHP, Node.js, Rails, Django, Spring, Laravel, Express 等）が複数記載されていれば合格寄りに判断
- フロントエンド系技術のみ（React, Vue.js, Angular, HTML/CSS, TypeScript 等）の場合はバックエンド不足と判断
- インフラ系技術（AWS, GCP, Azure, Docker, Kubernetes, Terraform 等）もバックエンド設計力の補強材料とする
- DB関連（MySQL, PostgreSQL, Redis, DynamoDB 等）の記載はスキーマ設計経験の傍証とする
- プロフィールの自己紹介文中に記載された技術スタックも同様に参照する

**明確な不合格ケース：**
- **SRE（Site Reliability Engineer）が主職種** → アプリケーション開発ではなくインフラ・運用領域が中心のため不合格
- **インフラエンジニア / プラットフォームエンジニアが主職種** → 同上
- **フロントエンド専門**でバックエンド経験がほとんどない（スキル欄にバックエンド技術が見当たらない）
- **ML（機械学習）エンジニア専門**でWebバックエンド開発経験がない場合 → MLとWebアプリケーション開発は領域が異なるため、Webバックエンドの経験が確認できなければ不合格寄りに判断
- バックエンド経験が浅い（1年未満、学習レベル）
- デザイナー、マーケター等エンジニア以外の職種が中心

バックエンドスキルが不足している、または初期設計を担えるレベルに達していないと判断できる場合は
「不合格」とし、具体的に何が不足しているか記録する。

### 判定の基本方針

- **積極的に判定を行い、判定保留は最終手段とする**
- リクマの情報（現職タイトル・会社名）＋LinkedInヘッドラインだけでも判定可能なケースが多い
- 例：リクマで「Engineering Manager」と表示されていれば要件3は合格と判断できる
- 例：リクマで「SREエンジニア テックリード」と表示されていればSRE専門として要件4は不合格と判断できる
- 確実に要件を満たさないと言える場合は「不合格」とする（判定保留にしない）
- 曖昧なケースでは合格寄りに判断する
- 「判定保留」は、リクマの役職情報もLinkedInヘッドラインも全く役職・スキルが読み取れない場合のみ

スクリーニング完了後、Slack MCP を使って `#times_morooka`（チャンネルID: `C08L355KPB5`）に通知を送る。

通知メッセージ例：
```
リクルートマーカーのスタジオテックリードスクリーニングが完了しました。
対象リスト: [リスト名]
対象人数: XX名 / 要件非該当: XX名 / 判定保留: XX名
```

## エラーハンドリング

- プロフィールページが閲覧できない場合（非公開等）→「プロフィール非公開のため判定不能」として記録
- リクルートマーカーのセッションが切れた場合 → ユーザーに再ログインを依頼
- LinkedIn でログインが求められた場合 → ユーザーに状況を伝え、対応を依頼
- LinkedIn でCAPTCHA等が表示された場合 → ユーザーに報告し、YouTrustでの確認に切り替え
