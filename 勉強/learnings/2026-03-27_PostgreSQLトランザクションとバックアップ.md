# PostgreSQL トランザクションとバックアップ
日付: 2026-03-27

## 学んだこと

### データベースの3原則
- 完全性：操作したデータが壊れない
- 一貫性（整合性）：システム間で操作したデータの整合性を保つ
- 可用性：必要なときにデータ操作ができること

### トランザクション処理
- トランザクション＝開始から終了まで実行した複数の処理をまとめて1つの取り引きと考える
- 全処理が成功→トランザクション成功（COMMIT）、1つでも失敗→全て破棄（ROLLBACK）
- 銀行の振込例：Aの口座から引く＋Bの口座に足す、片方だけ成功すると一貫性が壊れる

### トランザクションのしくみ（PostgreSQL実装）
- MVCCとWAL、トランザクションID（XID）とトランザクション状態を使ってCOMMITとROLLBACKを実現
- **COMMIT処理**：
  1. BEGIN：トランザクション開始、XIDを確保する
  2. データ操作：メモリ（共有バッファ）上にXID付きで新しい行データを作成＋WALバッファに変更データを積む
  3. COMMIT：WALにコミットレコード作成→WALバッファの変更データをWALディスクに同期→XIDのトランザクション状態をCOMMITEDにする
  - この時点ではメモリ上のデータはデータディスクに書き込まれない（checkpointerなどで後から書き込み）
- **ROLLBACK処理**：
  1. XIDのトランザクション状態をABORTEDにする
  2. COMMITと異なりWALディスクには同期処理されず、後からWALディスクに書かれる
  - XIDがCOMMITTEDでないデータは参照されないので、ABORTED状態のXIDのデータは外から見えなくなる
  - ABORTED状態のデータはVACUUM処理でデータディスクから削除される

### トランザクション分離レベル（PostgreSQL 3種類）
- **READ COMMITTED（デフォルト）**：文単位で一貫性を保証。クエリごとに最新の確定データを読むだけ。ロック保持時間が短く競合が起きにくい→並列処理に最も向いている。処理性能：高
- **REPEATABLE READ**：トランザクション開始時点の一貫性を保証。最初に見たスナップショットを保持し続ける。ほかの更新と衝突するとエラーになるため、リトライのコストがわずかに発生する。処理性能：中
  - 例：入札テーブルでトランザクション途中で商品参考価格が変わっても影響なし
- **SERIALIZABLE**：完全な直列化保証。全トランザクションの依存関係を監視。矛盾の予兆があれば即エラーにする。リトライ頻度が高く全体の処理量は落ちる。処理性能：低
  - 例：スタッフシフト管理で、同じ条件で別の行を更新した結果データ更新条件が崩れる危険がある場合にエラーで防ぐ

### READ COMMITTEDの具体例
- イベントテーブル（event_id, capacity, applicants）で「募集人数を超えなければ応募者数を加算」する処理
- Aさんのトランザクションが更新中（未コミット）→Bさんが同じ行を更新しようとするとロック待ち状態になる

### SERIALIZABLEの具体例
- スタッフシフトテーブルで「日中シフトを外す処理（ただし自分以外に日中シフトの人がいる場合に更新可能）」
- AさんとBさんが同時に実行→両方とも相手がまだシフトにいると見えるので両方更新成功→結果、日中シフトに穴が開く
- SERIALIZABLEにセットすることで、Aさんのコミット後にBさんのコミットはエラー（ロールバック）になる

### 可用性と安全性
- データベース単体で「可用性」を実現するのは難しい。クラッシュ時に操作の継続ができなくなる
- データベースを複数持つなどシステムに冗長性を持たせることで「可用性」を確保
- **レプリケーション（データ複製）**：各データベースがリアルタイムにデータを引き継ぎ、システム全体のダウンタイムを極力縮める
  - リードレプリカ：読取り専用データベース。参照負荷を分散できる。ただし更新負荷は分散できない（むしろ書き込みが増えるので更新負荷は増す）
  - 更新負荷が多いシステムでは、レプリケーションには可用性向上以上の効果は期待しないほうがよい
  - レプリケーションはバックアップにはならない（間違ってデータを削除したら全レプリカから削除される）

### バックアップの種類
- **フルバックアップ**：pg_dumpコマンドで手軽に実行可能。ただしデータが巨大になると時間的に難しくI/O負荷もかかる
- **差分バックアップ（PostgreSQL 18〜）**：最初にベースバックアップを取り、あとは差分バックアップで日々の差分だけを高速にバックアップ
  - 設定：`wal_level = replica`、`summarize_wal = on`（postgresql.conf）
  - 初回：`pg_basebackup -D backup/full -P`でフルバックアップ
  - 差分：`pg_basebackup --incremental=backup/前回/backup_manifest -D backup/YYYYMMDD`
  - 差分は前回のbackup_manifestを指定する必要がある
- **差分バックアップの統合**：`pg_combinebackup`コマンドで復元。フルバックアップ＋戻したい差分までを統合
  - `pg_combinebackup -o 出力ディレクトリ名 フルバックアップ 差分バックアップ1 差分バックアップ2 ...`
  - `pg_verifybackup`で統合した環境を検証
  - 出力ディレクトリをデータディレクトリに指定して`pg_ctl`で起動すれば復元したデータベースにアクセスできる
  - 任意の差分まで戻した状態でデータベースを復元できる
